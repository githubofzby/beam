# `beam` 仓库算法理解报告

本文档只描述当前仓库中仍保留的主线 merge 模式：

- `batch-ea`
- `batch-ea-blocked`

已删除的历史自研 baseline 不再作为当前代码路径的一部分，也不再作为本文档的分析对象。

## 仓库定位

当前 C++ 实现是一个扁平修正集图摘要原型，核心流程仍然是：

1. `Divide`
2. `Merge`
3. `Encode`
4. 可选 `Drop`
5. 可选写出 `G.txt / P.txt / Cp.txt / Cm.txt`

对应的核心文件：

- `src/main.cpp`
- `src/sweg.cpp`
- `include/sweg.hpp`
- `src/scoring_cpu.cpp`
- `src/cuda_scoring.cu`
- `tools/run_small_tests.py`

## 当前保留的 merge 模式

### `batch-ea`

`batch-ea` 是当前仓库的主线编码感知 merge 模式。它的关键流程是：

1. 在 divide 得到的每个 group 内构造 supernode 邻接聚合。
2. 用 encoding-aware proxy 生成候选对，并为每个 supernode 保留 top-`k` 候选。
3. 对候选对计算 exact local encoding gain。
4. 仅保留正 gain 候选；若启用 `--ea-use-threshold`，再按 gain ratio 过滤。
5. 在组内做无冲突 pair selection。
6. 立即执行 merge update。

### `batch-ea-blocked`

`batch-ea-blocked` 保持与 `batch-ea` 相同的局部编码增益目标，但改变执行调度：

1. 先准备多个 group 的候选与聚合表示。
2. 按 block 组织 scoring。
3. 每个 group 独立做无冲突 pair selection。
4. 将 merge update 延后到 block 末尾统一执行。

CPU 路径支持 blocked scheduling；CUDA 路径的说明和 `candidate_batch_budget` 控制也以该模式为准。

## 评分后端

- `--scoring-backend cpu`
  用于 `batch-ea` 和 `batch-ea-blocked`。

- `--scoring-backend cuda`
  当前主线 CUDA scoring 说明以 `batch-ea-blocked` 路径为准。

## 测试与验证

仓库自带的小图回归脚本会验证：

- `batch-ea`
- `batch-ea-blocked`

运行方式：

```bash
python3 tools/run_small_tests.py --binary build/beam
```

## 论文/实验建议

如果需要 baseline，对照实验建议使用仓库外部方法，例如 LDME，而不是恢复已删除的历史自研 merge 模式。
